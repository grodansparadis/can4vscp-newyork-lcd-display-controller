New York text display module

Used for LCD text displays

  • One or any number of similar length dispay lines.
  • More then one display scenaro shown in a round robin fashion.
  • Width.
  • Number of lines.
  • Display working. Stable, Round robin, button.
  • r - display register content. %ri001 display register 1.
  • p - display event content. Can contain sub event information. %piz,s (z=zone, s=subzone).
  • e - event information.

Thoughts

Charles

Using a message such as “Display Message”. The first byte would indicate if this is a completely resolved message ie. 0 indicates that the message has no variables whereas 1 indicates that a node is needed to interpret this.

Now, an intelligent LCD can process the unresolved message or can wait for a“regular expression” node to take this message and then send out another “Display Message”, this time with the parameter passing done and so byte 1 has a value of 0.

The question then is how should a message with parameters be formatted. One suggestion is to use either “regular expression” syntax or “xml” based syntax. We can work on a syntax for these messages.

Note also that the long messages can be specified by sending a “current segment/total segments” in one of the bytes so it is known before hand how many parts there are to the message.

Lastly we might need want to implement a “Multi-Part message available” and “Multi-part message request” to ensure only one node is sending a multi-part message at any one time so the segments don't get mixed up.

Ake

I think we need two classes. One for Level I and one for Level II. We can use a bit different thinking for the two. Users/implementors can chose the one they want to use.

On Level I some sort of multi event mechanism is a must.

Embedded parameters are fine. Some sort of formating capabilities needed here such as printf. Typically something like this is needed.

“High temperature alarm for zone %rb001. The temperature is %ri002 degrees Celisus.”

Where the construct %rb001 refers to a register with index 001 where the binary “b” data could be found and “i” is signed integer data and so on. This will be used to define display format in decision matrix rows for one of my modules (Text LCD module). The action argument decides which string to display and the escapes select where information is fetched for the displayed information. %cb1 is for instance the current event data byte 1 and is interpreted as a byte and so on.

In this a node can fill in values by itself.

Also something like

“High temperature alarm for zone %pb001. The temperature is %pi002 degrees Celisus.”

where %p is a parameter sent in a separate display event

and

“High temperature alarm for zone %e20,3. The temperature is %e10,2 degrees Celisus.”

where data is taken from an event class,type

Also lower level stuff like “move to row”, “clear” etc are needed.

Henk

I think a level 1 and a level 2 class device is a good idea.

Device is not intended for trouble shooting. It should only be used for indicating values and (alarm) status.

A generic smart display will be difficult to construct I think. In my opinion it will be easier to have displays with different firmware for each level.

example: Firmware for a display to control the heating wil be different from firmware that does security or air quality display functions.
